1. Введение

1.1. Назначение документа

Настоящий документ «Спецификация программных требований» (SRS) полностью описывает функциональные и нефункциональные требования к мобильному приложению «Запись на приём» для медицинских клиник, а также проектные ограничения и другие факторы, влияющие на разработку.

1.2. Границы и область применения

В данный SRS входят:
  • Требования к мобильному приложению «Запись на приём» для Android
Вне границ данного SRS остаются:
  • Реализация и архитектура серверной части (бэкенд), а также API для интеграции с МИС и внешними сервисами описываются в отдельном документе. В данном SRS указываются только требования к взаимодействию с ними.

1.3. Целевая аудитория и использование документа

Документ предназначен для:

  1. Заказчика (для подтверждения соответствия продукта ожиданиям)
  2. Разработчиков (для реализации функционала)
  3. Тестировщиков (для создания тест-кейсов и проверки)
  4. Технических писателей (для создания пользовательской документации)

2. Общее описание

2.1. Перспектива системы или продукта

Приложение «Запись на приём» является самостоятельным ПП, предназначенным для установки на мобильные устройства под управлением ОС Android.

2.2. Функции системы или продукта

2.2.1. Навигация и информация о клинике

  1. Отображение списка клиник (если сеть) с указанием адреса и расстояния до пользователя
  2. Показывает путь до выбранной клиники (интеграция с Яндекс.Картами)
  3. Отображение рейтинга клиники (интеграция с сервисом «ПроДокторов»)

2.2.2. Управление профилем

  1. Заполнение профиля (ФИО, полис, дата рождения)
  2. Привязка профилей членов семьи

2.2.3. Выбор пациента для записи

  1. Выбор «Для себя» или «Для другого» (из списка привязанных профилей или добавление нового)

2.2.4. Поиск и выбор врача

  1. Фильтрация по специальности/стажу/отзывам
  2. Добавление врачей в «Избранное»
  3. Отображение проверенных отзывов (интеграция с сервисом «ПроДокторов»)

2.2.5. Запись к врачу (включает в себя):

  1. Выбор доступной даты и времени (календарь с ближайшим слотом)
  2. Выбор формата приёма (онлайн / офлайн)
  3. Поле «Комментарий для клиники» (для особых потребностей)
  4. Подтверждение записи с отображением номера
  5. Напоминания (push/SMS/e-mail с настраиваемым временем и ссылкой на видеозвонок, если формат онлайн)

2.2.6. Управление записью

  1. Отмена записи
  2. Перенос записи с автоматическим подбором свободных слотов
  3. Смена пациента в существующей записи

2.3. Характеристики пользователей

Пользователи смартфонов на Android, нуждающиеся в записи к врачу.

2.4. Ограничения, допущения и зависимости

Ограничения:

  1. Приложение работает при наличии интернет-соединения (для получения актуальных данных)
  2. Поддерживаемая ОС: Android 8.0 и выше
  3. Размер экрана: не менее 4.5 дюймов

Допущения:

  1. Пользователь дал разрешение на доступ к геолокации (для построения пути до клиники)
  2. Устройство имеет доступ к интернету

Зависимости:

  1. Работоспособность функций навигации зависит от доступности и корректной работы API Яндекс.Карт
  2. Функция отображения проверенных отзывов зависит от стабильной работы API сервиса «ПроДокторов»
  3. Функция онлайн-записи зависит от доступности API интеграционной платформы «Нетрика» и готовности медицинской информационной системы (МИС) клиники принимать данные
  4. Актуальность расписания врачей и записи зависит от своевременного обновления данных в МИС клиники

2.5. Операционная среда

  1. ОС: Android 8.0 и выше
  2. Устройства: смартфоны с размером экрана ≥ 4.5 дюймов
  3. Сеть: Wi-Fi или мобильные данные (3G/4G/5G)

3. Функциональные требования

3.1. Группа: Навигация и информация о клинике

ФТ001. Приложение должно определять геолокацию пользователя (с его разрешения) для отображения ближайших клиник в списке.
ФТ002. Приложение должно отображать маршрут от текущего местоположения пользователя до выбранной клиники на карте (интеграция с Яндекс.Картами) и показывать ориентировочное время в пути.

3.2. Группа: Управление профилем

ФТ003. Приложение должно предоставлять возможность заполнения профиля пользователя: ФИО, номер полиса ОМС/ДМС, дата рождения и т.д.
ФТ004. Приложение должно позволять добавлять, редактировать и удалять профили членов семьи.

3.3. Группа: Выбор пациента для записи

ФТ005. Приложение должно предоставлять выбор «Для себя» / «Для другого» перед началом выбора врача.
ФТ006. При выборе «Для другого» приложение должно отображать список добавленных профилей членов семьи и давать возможность выбрать нужный.

3.4. Группа: Поиск и выбор врача

ФТ007. Приложение должно отображать список врачей с возможностью фильтрации по специальности, стажу работы и рейтингу (по отзывам).
ФТ008. Приложение должно предоставлять карточку врача с подробной информацией: ФИО, специальность, стаж, рейтинг, список проверенных отзывов (интеграция с сервисом «ПроДокторов»).
ФТ009. Приложение должно позволять добавлять врачей в «Избранное» и удалять из избранного.
ФТ010. Приложение должно отображать отдельный экран «Избранное» со списком сохранённых врачей.

3.5. Группа: Запись к врачу

ФТ011. Приложение должно отображать календарь с доступными слотами для выбранного врача. Недоступные слоты должны быть заблокированы.
ФТ012. Приложение должно автоматически подсвечивать ближайший доступный слот для записи.
ФТ013. Приложение должно позволять пользователю выбрать формат приёма: онлайн (видеоконсультация) или офлайн (визит в клинику).
ФТ014. При выборе онлайн-формата приложение должно показывать информационный чек-лист «Когда подходит онлайн-приём».
ФТ015. Приложение должно содержать поле «Комментарий для клиники» для указания особых потребностей (например, необходимость инвалидной коляски).
ФТ016. Приложение должно отображать экран подтверждения записи со всеми выбранными данными (пациент, врач, дата, время, формат, комментарий).
ФТ017. После подтверждения записи приложение должно показывать экран успеха с уникальным номером записи и отправлять in-app уведомление.

3.6. Группа: Уведомления и напоминания

ФТ018. Приложение должно отправлять push/SMS/e-mail напоминания о предстоящем приёме за 1 день и за 1 час до визита (время напоминания должно быть настраиваемым).
ФТ019. В напоминании для онлайн-приёма должна быть прямая ссылка на видеозвонок.
ФТ020. Приложение должно предоставлять кнопку «Добавить в календарь» в напоминании для сохранения события в календарь ОС.

3.7. Группа: Управление записью (отмена / перенос)

ФТ021. Приложение должно позволять отменять запись из раздела «Мои приёмы» (предстоящие) с запросом подтверждения.
ФТ022. Приложение должно позволять переносить запись на другую дату и время с автоматическим подбором ближайших свободных слотов.
ФТ023. Приложение должно предоставлять возможность сменить пациента в существующей записи (если пользователь ошибся при выборе).

3.8. Группа: Просмотр истории

ФТ024. Приложение должно отображать список предстоящих приёмов и историю прошедших приёмов (две вкладки в разделе «Мои приёмы»).

4. Нефункциональные требования

4.1. Требования к производительности

НФТ001. Отображение экранов (список врачей, календарь, подтверждение записи) после загрузки данных должно занимать не более 5 секунд при стабильном интернет-соединении.
НФТ002. Построение маршрута до клиники (интеграция с Яндекс.Картами) должно отображаться не более, чем через 2 секунды после нажатия кнопки «Построить маршрут».
НФТ003. Приложение должно потреблять не более 50 МБ оперативной памяти в фоновом режиме и не более 150 МБ при активном использовании (карта, календарь).

4.2. Требования к надёжности и доступности

НФТ004. Приложение должно сохранять работоспособность при обрыве интернет-соединения: показывать кэшированные данные (список врачей, последние отзывы) и уведомлять пользователя об отсутствии соединения.
НФТ005. Система должна работать без перебоев не менее 99,9% времени.
НФТ006. Система должна быть доступна пользователям из любой точки России.

4.3. Требования к удобству использования

НФТ007. Все обязательные поля для заполнения профиля (ФИО, полис, дата рождения) должны быть явно помечены звёздочкой.
НФТ008. Экран должен поддерживать вертикальную прокрутку для просмотра списков (врачей, клиник).
НФТ009. Переключение между вкладками «Для себя» / «Для другого», «Предстоящие» / «Прошедшие», «Свой профиль» / «Профиль членов семьи» должно происходить одним нажатием (без перезагрузки экрана).
НФТ010. При вводе данных в полях (например, серия/номер полиса) должны отображаться подсказки по формату ввода.
НФТ011. При выборе даты и времени в календаре недоступные слоты должны быть визуально заблокированы (серым цветом) и не реагировать на нажатие.
НФТ012. Приложение должно поддерживать адаптивный дизайн для экранов от 4.5 до 7 дюймов, все элементы должны быть читаемы без масштабирования.

4.4. Требования к сопровождаемости и совместимости

НФТ013. Приложение должно корректно запускаться и работать на всех устройствах с ОС Android версии 8.0 и выше.
НФТ014. Приложение должно корректно отображаться на мобильных устройствах с размером экрана не менее 4.5 дюймов.

5. Требования к внешним интерфейсам

5.1. Пользовательские интерфейсы

  • Главный экран: новости и акции
  • Экран «Записаться на приём»: список врачей с фильтрацией по специальности, стажу, отзывам (интеграция с сервисом «ПроДокторов»). При нажатии на врача открывается «Карточка врача»
  • Экран «Карточка врача»: отображает ФИО врача, специальность, стаж, рейтинг, отзывы
  • Экран «Мои приёмы»: две вкладки: «Предстоящие» – содержит список предстоящих приёмов, «Прошедшие» – содержит список прошедших приёмов (история). Обе вкладки с фильтрацией по пользователю
  • Экран «Профиль»: две вкладки: «Свой профиль» – содержит данные пользователя. «Профиль членов семьи» – содержит список профилей членов семьи
  • Экран «Клиники»: список клиник. При нажатии на клинику открывается «Карточка клиники»
  • Экран «Карточка клиники»: отображает фотографии клиники, часы работы, контакты, расположение на интерактивной карте (интеграция с Яндекс.Картами)
  • Экран «Избранное»: список врачей, добавленных в избранное. При нажатии на врача открывается его карточка

5.2. Аппаратные интерфейсы

  • Приложение должно использовать GPS для определения геолокации
  • Приложение должно поддерживать мультитач-взаимодействие через сенсорный экран

5.3. Программные интерфейсы

Наименование интерфейса Назначение Формат данных Протокол
API «Нетрика (N3)» Запись на приём, получение расписания врачей, доступных слотов, создание/отмена записи JSON HTTPS
API «ПроДокторов» Получение рейтингов, отзывов о врачах и клиниках JSON HTTPS
Yandex MapKit SDK Отображение клиник на карте, построение маршрута, геокодирование - HTTPS
Android Calendar Provider Добавление записи на приём в системный календарь устройства - -

5.4. Коммуникационные интерфейсы

  • Протокол: HTTPS
  • Формат запросов: REST GET с параметрами
  • Кодировка: UTF-8

6. Прочие требования

БТ001. Количество пользователей приложения «Запись на приём» должно увеличиться к концу года на 30%.
БТ002. Средний рейтинг приложения в Google Play после первого месяца эксплуатации – не ниже 4.3.
БТ003. Доля успешных записей на приём (без ошибок) – не менее 98%.

7. Верификация и валидация

7.1. Критерии приёмки

  1. Приложение успешно проходит все функциональные тесты
  2. Время отображения информации не превышает 5 секунд
  3. Приложение работает на Android 8.0+ и экранах ≥ 4.5"
  4. Геолокация определяет местоположение при каждом запуске
